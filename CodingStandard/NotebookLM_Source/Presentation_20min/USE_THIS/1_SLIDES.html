§1 · Required

Project Structure 3-Layer Architecture — 10 criteria (7 required + 3 recommended)

{SolutionName}/ ├── .editorconfig ├── {SolutionName}.sln ├── src/ │ └── {ProjectName}/ │ ├── Controllers/ ← API Endpoints │ ├── Services/ ← Business Logic │ │ └── Interfaces/ │ ├── DataAccess/ ← Dapper + Repository │ │ ├── Connections/ │ │ └── Repositories/ │ │ └── Interfaces/ │ ├── Models/ │ │ ├── Entities/ ← DB Mapping │ │ └── Requests/ ← Input DTOs │ ├── Filters/ │ └── Validators/ └── tests/ ← Separate from src/ └── UnitTest/
  • 1.1 Separate into 3 Layers: Controllers, Services, DataAccess
  • 1.2 Models/ must have Entities/ and Requests/ sub-folders
  • 1.3 Services/ must have Interfaces/ (Dependency Inversion)
  • 1.4 DataAccess/ must have Connections/ + Repositories/
  • 1.5 Repositories/ must have Interfaces/ (Mockable)
  • 1.6 Filters/ — separate from Controller
  • 1.7 tests/ must be separate from src/

§3 · Required — 38 criteria

DataAccess Layer Dapper + Repository Pattern — SQL injection prevention, async, transactions

Pattern Requirements

  • 3.1 Dapper only — no EF Core, NHibernate
  • 3.2 Repository Pattern — no IDbConnection in Service
  • 3.3 IDbConnectionFactory — no new SqlConnection()
  • 3.4 BaseRepository — generic wrapper for Dapper

🛡️ SQL Security

  • 3.10 Always close with using CWE-404
  • 3.11 No SELECT * — specify columns CWE-200

§4 · §5 · Required

Service & Controller Business Logic + Thin API Layer — 16 criteria

§4 Service Layer

  • 4.1 Every Service must have an Interface
  • 4.2 Constructor Injection only — no new
  • 4.4 async + CancellationToken for all I/O
  • 4.5 Guard Clause at method start
  • 4.6 try-catch + structured logging
  • 4.7 Return Response DTO — never return Entity
  • 4.8 No hard-coded values — use IOptions<T>

§5 Controller Layer

  • 5.1 Inherit ControllerBase (not Controller)
  • 5.2 [ApiController], [Route], [Produces]
  • 5.3 Inject IService — no business logic
  • 5.4 [Authorize] on every protected endpoint
  • 5.7 Controller = Receive → Call → Return
  • 5.8 Pass CancellationToken to Service
💡
Defense-in-Depth: Guard Clause (§4.5) + FluentValidation (§15) work together per OWASP Input Validation Cheat Sheet

§8 · §9 · Required

Error Handling & Config Logging, unified responses, per-environment config — 13 criteria

§8 Error Handling

  • 8.1 Every Service method: try-catch
  • 8.2 Log Error + context (Method, Parameters)
  • 8.4 Use ILogger<T> — never Console.WriteLine
  • 8.5 Unified response: Status, IsSuccess, Message
  • 8.6 Never return StackTrace to client
  • 8.7 LogFilter for auto request/response logging
JSON — Unified Response
{ "status": 500, "isSuccess": false, "message": "Internal server error." }

§9 Configuration

  • 9.1 appsettings.json = base configuration
  • 9.2 At least 3 environments: Dev, UAT, Prod
  • 9.3 Connection strings in appsettings.{Env}.json
  • 9.4 Secrets never in source code
  • 9.6 Response messages in config — not hard-coded

§16.1 — Structured Logging: Use template parameters, never string concatenation.

§15 · Required — 10 criteria

Input Validation "Never trust input" — FluentValidation as primary validation framework

  • 15.1 Every request model must have validation
  • 15.2 Validate at Controller level before Service
  • 15.3 Required, Length, Range, Email, Date, Regex
  • 15.4 Custom validation → separate Validator class
  • 15.6 Trim() all string inputs before use
  • 15.7 Never accept dynamic / object
  • 15.8 Pagination: pageSize 1–100, pageNumber ≥ 1
C# — FluentValidation
public class CreateCustomerRequestValidator : AbstractValidator<CreateCustomerRequest> { public CreateCustomerRequestValidator() { RuleFor(x => x.CustomerName) .NotEmpty() .MaximumLength(200); RuleFor(x => x.Email) .NotEmpty() .EmailAddress(); RuleFor(x => x.PhoneNumber) .Matches(@"^\d{9,10}$") .When(x => x.PhoneNumber != null); } }

§12 · Required — 15 criteria

Testing Standards xUnit + Moq — AAA Pattern — Coverage gates

  • 12.1 Every Service class must have Unit Test
  • 12.2 Framework: xUnit
  • 12.3 Mocking: Moq or NSubstitute
  • 12.8 Naming: {Method}_{Scenario}_{Expected}
  • 12.9 AAA Pattern: Arrange → Act → Assert
  • 12.10 No real DB — use Mock Repository
  • 12.11 Coverage ≥ 80% Service, ≥ 60% Overall
  • 12.12 Tests run in CI automatically
  • 12.14 Every bug fix → Regression Test
C# — xUnit + Moq + AAA
[Fact] public async Task GetCustomerAsync_WhenFound_Returns200() { // Arrange var entity = new CustomerEntity { CustomerId = "C001", CustomerName = "John" }; _mockRepo .Setup(r => r.GetByIdAsync("C001", It.IsAny<CancellationToken>())) .ReturnsAsync(entity); // Act var result = await _sut.GetCustomerAsync("C001"); // Assert Assert.True(result.IsSuccess); Assert.Equal(200, result.Status); }

§13 · Required — 12 criteria

API Design & Versioning RESTful conventions, pagination, ISO 8601 datetime

  • 13.1 URL Pattern: /api/v{n}/{resource}
  • 13.2 HTTP Methods: GET, POST, PUT, DELETE
  • 13.3 Status Codes per RFC 9110
  • 13.4 Collections: Pagination (default 20, max 100)
  • 13.8 JSON = camelCase
  • 13.9 DateTime = ISO 8601 UTC
  • 13.10 No verbs in URL — use HTTP Methods
  • 13.11 Strongly-typed DTO for every endpoint

§17 · §18 · Required

Performance & HTTP Client P95 ≤ 500ms — IHttpClientFactory + Polly — 20 criteria

§17 Performance

  • 17.1 Response: P95 ≤ 500ms (CRUD)
  • 17.2 No N+1 Query — use JOIN/Batch
  • 17.3 Every query must use Index
  • 17.4 Payload ≤ 1 MB
  • 17.5 Async/Await throughout pipeline
  • 17.6 No .Result, .Wait()
  • 17.7 Dispose with using

§18 HTTP Client

  • 18.1 IHttpClientFactory — never new HttpClient()
  • 18.3 Timeout on every client (30s default)
  • 18.4 Retry Policy (Polly) for 5xx
  • 18.5 Exponential Backoff: 1s → 2s → 4s (max 3)
  • 18.6 Circuit Breaker: 5 failures / 30s
  • 18.8 Send X-Correlation-Id on every call

§16 · Required — 10 criteria

Observability & Monitoring Structured logging, tracing, health checks, alerting

📋
Structured Logging
  • JSON format in production
  • CorrelationId on every request
  • Correct Log Levels
  • No PII / sensitive data
🔍
Distributed Tracing
  • OpenTelemetry or App Insights
  • X-Correlation-Id header
  • Log outbound HTTP calls
  • URL, Method, Status, Duration
❤️
Health & Metrics
  • Self (liveness) check
  • DB (readiness) check
  • /metrics endpoint
  • P50/P95/P99 + Error Rate
🚨
§16.9: Critical errors must trigger an alert mechanism — no silent failures in production.

  • 19.1 Pin exact version — no wildcards
  • 19.2 Approved Package List — review required

  • K8s manifests (deployment.yaml)
