## 1. บทสรุปสำหรับผู้บริหาร (Executive Summary)

เอกสารนี้จัดทำขึ้นเพื่อวิเคราะห์และนำเสนอแนวทางการแก้ไขระบบตาม Requirement ใหม่ที่ต้องการให้ลูกค้าประเภท **AMB123** ที่ทำการนัดหมายเพื่อยื่นเอกสาร ได้รับการตอบกลับเป็น `HTTP Status 203 Non-Authoritative Information` โดยมีเงื่อนไขสำคัญคือ **ห้ามมีการแก้ไขโค้ดของแอปพลิเคชันหลัก (OMNICHANNEL_API)**

จากการวิเคราะห์ทางเลือกทั้งหมด แนวทางที่ได้รับเลือกคือ **การใช้ API Gateway/Reverse Proxy** ในการจัดการเงื่อนไขดังกล่าว ซึ่งเป็นวิธีเดียวที่สอดคล้องกับข้อบังคับทั้งหมด และยังเป็นสถาปัตยกรรมที่เหมาะสมตามหลักปฏิบัติสากล

## 2. Requirement

- **เงื่อนไข:** หากลูกค้าเป็นประเภท `AMB123`
- **Action:** เมื่อทำการ "นัดยื่นเอกสาร" ให้ระบบตอบกลับด้วย `HTTP Status 203`
- **ข้อบังคับ:** ห้ามแก้ไขโค้ดของแอปพลิเคชัน `OMNICHANNEL_API`

## 3. ทางเลือกที่พิจารณา (Options Analysis)

เราได้พิจารณาทางเลือกที่เป็นไปได้ทั้งหมด 3 แนวทาง โดยมีรายละเอียดและผลการประเมินดังนี้

| # | Solution | วิธีการ | ข้อดี | ข้อเสีย | คะแนน |
|---|----------|---------|-------|---------|--------|
| **1** | **API Gateway / Reverse Proxy** | ดักจับ Request ที่ Gateway และคืนค่า `203` หากเงื่อนไขตรง โดยไม่ส่งต่อให้แอปพลิเคชัน | - ไม่ต้องแก้โค้ด 100%<br>- แยก Logic ชัดเจน<br>- จัดการง่ายในอนาคต | - ต้องมี Gateway ในระบบ | **10/10** |
| **2** | **Middleware ในแอปพลิเคชัน** | เพิ่มโค้ด Middleware เพื่อตรวจสอบและคืนค่า `203` ก่อนถึง Logic หลัก | - Logic รวมอยู่ในที่เดียว | - **ขัดแย้งกับข้อบังคับหลัก** (ต้องแก้โค้ด) | **1/10** |
| **3** | **Configuration-based Logic** | แก้โค้ดเพื่อสร้างกลไกอ่าน Rule จากภายนอก (Config file/DB) | - ยืดหยุ่นในระยะยาว | - **ขัดแย้งกับข้อบังคับหลัก** (ต้องแก้โค้ดครั้งใหญ่)<br>- Over-engineer | **1/10** |

## 4. Solution ที่ได้รับเลือกและเหตุผล

**Solution ที่เลือก:** **Option 1: การใช้ API Gateway / Reverse Proxy**

**เหตุผลในการเลือก:**

1.  **สอดคล้องกับข้อบังคับ 100%:** เป็นแนวทางเดียวที่ไม่ต้องมีการแก้ไขโค้ดของแอปพลิเคชัน `OMNICHANNEL_API` เลยแม้แต่บรรทัดเดียว
2.  **Low Impact & Low Risk:** การเปลี่ยนแปลงเกิดขึ้นที่ระดับโครงสร้างพื้นฐาน (Infrastructure) ซึ่งแยกขาดจาก Business Logic ทำให้ไม่มีผลกระทบข้างเคียง (Side Effect) ต่อการทำงานของระบบส่วนอื่นๆ
3.  **Architecturally Sound:** การจัดการ Logic การ Routing, การแปลง Request/Response หรือเงื่อนไขพิเศษที่ระดับ Gateway เป็นหลักการออกแบบสถาปัตยกรรมสมัยใหม่ (Modern Architecture) ที่ดี ช่วยลดภาระของแอปพลิเคชันหลัก (Single Responsibility Principle)
4.  **ง่ายต่อการจัดการและ Rollback:** การเปิด/ปิด หรือแก้ไขเงื่อนไขนี้ในอนาคต สามารถทำได้โดยการแก้ไขการตั้งค่า (Configuration) ของ Gateway เท่านั้น ซึ่งทำได้รวดเร็วและสามารถยกเลิกการเปลี่ยนแปลง (Rollback) ได้ทันทีหากเกิดปัญหา

ด้วยเหตุผลทั้งหมดนี้ แนวทางการใช้ API Gateway จึงเป็นทางเลือกที่เหมาะสม มีประสิทธิภาพ และปลอดภัยที่สุดสำหรับ Requirement นี้
