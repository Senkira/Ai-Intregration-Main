การกำหนด Coding Standards ระดับองค์กรผ่าน Gemini Code Assist Extension บน VS Code
Version 1.1 5 March 2026 Confidential — Internal Use Only

1. ขอบเขตของเอกสาร

ข้อกำหนดสำคัญ: ทุก Solution ในเอกสารนี้ทำงานผ่าน Gemini Code Assist Extension บน VS Code เท่านั้น ไม่มี Solution ใดต้องใช้ extension ภายนอก, proxy server หรือ wrapper application ใด ๆ ทั้งสิ้น

เอกสารฉบับนี้นำเสนอแนวทางในการกำหนด prompt หรือ context ระดับองค์กร ให้ถูกเรียกใช้ทุกครั้งเมื่อ developer ทำงานกับ Gemini Code Assist โดยมีเป้าหมายให้ผู้ใช้ปลายทางไม่สามารถแก้ไขหรือมองเห็นเนื้อหาดังกล่าวได้

ข้อมูลทั้งหมดตรวจสอบจาก Google Cloud Official Documentation (อัปเดตล่าสุด 5 March 2026)

2. ปัญหาที่ต้องการแก้ไข

  • ต้องการให้ทีมทั้งองค์กรปฏิบัติตาม Coding Standards เดียวกัน
  • ต้องการ prompt/context ที่ผู้ใช้ไม่สามารถแก้ไขได้ หรือไม่ต้องเห็นเนื้อหาโดยตรง
  • ใช้ Gemini Code Assist (Standard หรือ Enterprise) ผ่าน VS Code
  • ต้องการจัดการแบบรวมศูนย์และ deploy ได้ทั่วทั้งองค์กร

3. ข้อเท็จจริงที่ต้องทราบก่อน

ข้อจำกัดของ Gemini Code Assist ณ ปัจจุบัน:

5. Solution A — Code Customization (Enterprise Only)

Enterprise — แนะนำเป็นอันดับแรก

5.1 หลักการทำงาน

Code Customization เป็น feature ของ Gemini Code Assist Enterprise ที่เชื่อมต่อ private repository ขององค์กร (GitHub, GitLab, Bitbucket) เข้ากับ Google Cloud จากนั้น Gemini จะ index เนื้อหาใน repository และนำ coding style, internal API, patterns, documentation ไปใช้เป็น context ใน code completion, code generation และ chat response โดยอัตโนมัติ

อ้างอิง: Code Customization Overview

Private Repository
(Coding Standards)
→
Developer Connect
(Google Cloud)
→
Indexing Service
(Re-index ทุก 24 ชม.)
→
Gemini Code Assist
(VS Code Extension)

5.2 เหตุผลที่แนะนำ

  • ผู้ใช้ไม่เห็น "prompt" โดยตรง — context มาจาก indexed data ไม่ใช่ plain text ที่ตั้งค่าใน IDE
  • ผู้ใช้แก้ไข index ไม่ได้ — Admin ควบคุมผ่าน IAM roles บน Google Cloud Console
  • ทำงานผ่าน Gemini Code Assist extension โดยตรง ไม่ต้องติดตั้งอะไรเพิ่ม
  • ครอบคลุมทั้ง code completion, code generation (GA ก.ย. 2024) และ chat (GA เม.ย. 2025)
  • Auto re-index ทุก 24 ชั่วโมง เมื่อ push การเปลี่ยนแปลงเข้า repository

5.3 Prerequisites

  • Gemini Code Assist Enterprise subscription
  • Gemini Code Assist VS Code extension version 2.18.0+
  • Developer Connect ใน supported region: us-central1, europe-west1, asia-southeast1
  • Repository hosted บน GitHub.com, GitHub Enterprise Cloud, GitHub Enterprise Server, GitLab.com, GitLab Enterprise, Bitbucket Cloud หรือ Bitbucket Data Center
  • IAM roles:
    • Admin: roles/cloudaicompanion.codeRepositoryIndexesAdmin + roles/cloudaicompanion.user
    • Developer: roles/cloudaicompanion.repositoryGroupsUser + roles/cloudaicompanion.user

5.4 ขั้นตอนการตั้งค่า

  1. สร้าง Private Repository สำหรับ Coding Standards รวบรวม coding standards, design patterns, code examples และ documentation 
    company-coding-standards/
├── README.md
├── coding-standards/
│   ├── java-standards.md
│   ├── python-standards.md
│   └── dotnet-standards.md
├── architecture/
│   ├── design-patterns.md
│   └── security-best-practices.md
├── examples/
│   ├── java-samples/
│   └── dotnet-samples/
└── templates/
    └── project-templates/
  2. ตั้งค่า Developer Connect เชื่อมต่อ repository provider กับ Google Cloud project (Developer Connect Documentation)
  3. สร้าง Code Repository Index 
    gcloud gemini code-repository-indexes create INDEX_NAME \
    --project=PROJECT_ID \
    --location=REGION

    ใช้เวลาประมาณ 30 นาที ถึง 1 ชั่วโมง จำกัด 1 index ต่อ project และต่อ organization

  4. สร้าง Repository Group และเพิ่ม repository 
    gcloud gemini code-repository-indexes repository-groups create REPO_GROUP \
    --project=PROJECT_ID \
    --location=REGION \
    --code-repository-index=INDEX_NAME \
    --repositories='[{"resource": "REPO_RESOURCE_NAME", "branchPattern": "main"}]'
  5. กำหนดสิทธิ์ IAM ให้ Developer 
    gcloud projects add-iam-policy-binding PROJECT_ID \
    --member='group:developers@company.com' \
    --role='roles/cloudaicompanion.repositoryGroupsUser'
  6. ตรวจสอบสถานะ Indexing ดู log ผ่าน Logs Explorer บน Google Cloud Console

5.5 สถานะที่ผู้ใช้จะเห็นใน VS Code

สถานะความหมาย
"All set. Code customization is enabled and configured."พร้อมใช้งาน
"Unavailable"ผู้ใช้ยังไม่มีสิทธิ์ หรือ repository group ว่าง
"Unset"Admin ยังไม่ได้เปิดใช้งาน

5.6 การใช้ Remote Repository Context ใน Chat

ผู้ใช้สามารถพิมพ์ @REPOSITORY_NAME ใน chat เพื่อเจาะจง indexed repository เป็น primary context:

// ตัวอย่างใน Gemini Code Assist chat:
@company-coding-standards Refactor this code to follow our conventions.
@company-coding-standards What is the standard error handling pattern?

ข้อดี

  • ผู้ใช้ไม่เห็นเนื้อหา prompt โดยตรง
  • Admin ควบคุม index ผ่าน IAM
  • ทำงานผ่าน Gemini extension โดยตรง
  • ครอบคลุม code completion + chat
  • Auto re-index ทุก 24 ชม.
  • ผู้ใช้ไม่ต้องตั้งค่าอะไร

ข้อจำกัด

  • ต้องใช้ Enterprise license (~$45-54/user/month)
  • Developer Connect region จำกัด 3 regions
  • Index สูงสุด 20,000 repositories, 500 groups/index, 500 repos/group
  • Indexing ครั้งแรกอาจใช้เวลานาน (re-index ทุก 24 ชม.)
  • ไม่ใช่ "system prompt" แต่เป็น context จาก indexed code

6. Solution B — Rules via Workspace Settings

Standard / Enterprise

6.1 หลักการทำงาน

Gemini Code Assist มี feature ชื่อ Rules (Settings > Extensions > Gemini Code Assist > Rules) เมื่อตั้งค่าแล้ว rule จะถูก include ในทุก chat prompt โดยอัตโนมัติ

การกำหนด rules ผ่าน .vscode/settings.json ที่ commit เข้า Git ทำให้ทุกคนที่ clone โปรเจกต์จะได้ rules เดียวกัน

อ้างอิง: Chat with Gemini Code Assist — Create Rules

6.2 ขั้นตอนการตั้งค่า

  1. สร้างไฟล์ .vscode/settings.json ในโปรเจกต์ 
    {
  "geminicodeassist.rules": "Always follow these coding standards:\n1. Use descriptive variable names\n2. Include proper error handling for every method\n3. Add documentation comments for public methods\n4. Follow SOLID principles\n5. Maximum method length is 30 lines\n6. Always generate unit tests when creating new functions"
}
  2. Commit และ push ไฟล์เข้า Git 
    git add .vscode/settings.json
git commit -m "Add organization coding rules for Gemini Code Assist"
git push
  3. ทีมทุกคน pull repository — จะได้ workspace settings ทันที
ข้อจำกัดสำคัญ — ผู้ใช้เห็นและแก้ไขเนื้อหาได้:
  • ผู้ใช้สามารถเปิด .vscode/settings.json หรือ Settings UI แล้วอ่าน rules ได้
  • ผู้ใช้สามารถ override ด้วย User Settings ซึ่งมี priority สูงกว่า Workspace Settings
  • VS Code ไม่มีกลไกล็อค settings

6.3 มาตรการเสริมเพื่อลดการแก้ไข

  • กำหนดนโยบายองค์กรห้ามแก้ไข workspace settings
  • ใช้ CI/CD pipeline ตรวจสอบว่า .vscode/settings.json ตรงกับ template
  • ใช้ Git pre-commit hook reject การแก้ไข settings file

7. Solution C — Custom Commands via Workspace Settings

Standard / Enterprise

7.1 หลักการทำงาน

Gemini Code Assist รองรับ Custom Commands ที่เรียกจาก Quick Pick menu (Ctrl+I) ตั้งค่าผ่าน workspace settings เพื่อให้ทีมทุกคนมี command มาตรฐานเดียวกัน

อ้างอิง: Chat with Gemini Code Assist — Create Custom Commands

7.2 ขั้นตอนการตั้งค่า

  1. เพิ่ม Custom Commands ใน .vscode/settings.json 
    {
  "geminicodeassist.customCommands": {
    "review-standard": "Review this code against our coding standards: proper error handling, SOLID principles, max 30 lines per method, descriptive naming",
    "generate-test": "Generate unit tests following AAA pattern, meaningful test names, edge cases coverage",
    "add-docs": "Add documentation comments following our standard format"
  }
}
  2. Commit และ push — ทีม pull ไปใช้ได้ทันที

7.3 การใช้งานฝั่งผู้ใช้

ผู้ใช้กด Ctrl+I แล้วเลือก command เช่น review-standard จาก Quick Pick menu

ข้อจำกัดเช่นเดียวกับ Solution B: ผู้ใช้สามารถเห็นเนื้อหา command ได้ใน Settings และ override ด้วย User Settings ได้

9. แผนการดำเนินงาน

Phase 1 — Preparation (สัปดาห์ที่ 1)

  • รวบรวม Coding Standards ขององค์กร
  • จัดทำ documentation, code examples และ templates
  • สร้าง private repository สำหรับ standards
  • ตรวจสอบ Gemini Code Assist Enterprise license

Phase 2 — Setup (สัปดาห์ที่ 2)

  • ตั้งค่า Developer Connect บน Google Cloud
  • สร้าง Code Repository Index และ Repository Group
  • กำหนดสิทธิ์ IAM สำหรับ developer
  • สร้าง .vscode/settings.json template พร้อม rules และ custom commands

Phase 3 — Pilot (สัปดาห์ที่ 3)

  • ทดสอบกับ pilot group (5-10 คน)
  • ตรวจสอบคุณภาพ code suggestions
  • รวบรวม feedback และปรับปรุง standards repository

Phase 4 — Rollout (สัปดาห์ที่ 4)

  • Deploy ทั้งองค์กร
  • จัดทำ training สำหรับ developer
  • Monitor usage ผ่าน Google Cloud Console

11. คำถามที่พบบ่อย

Q: มีวิธีซ่อน prompt จากผู้ใช้ 100% ได้หรือไม่?

Gemini Code Assist ไม่มี "hidden system prompt" feature โดยตรง แต่ Code Customization (Enterprise) เป็นทางเลือกที่ใกล้เคียงที่สุด เพราะ context มาจาก indexed repository ซึ่งผู้ใช้ไม่เห็นว่า data ถูก inject อย่างไร

Q: ถ้าใช้ Standard license จะทำได้อย่างไร?

ใช้ Solution B (Rules) + Solution C (Custom Commands) ผ่าน workspace settings แต่ต้องยอมรับว่าผู้ใช้สามารถเห็นและ override เนื้อหาได้

Q: Code Customization ต่างจาก Rules อย่างไร?

Code Customization: ทำงานที่ระดับ model — Gemini search indexed repository แล้วรวม relevant snippets เข้า context ก่อนส่งไป model ครอบคลุมทั้ง code completion, generation และ chat
Rules: ทำงานที่ระดับ chat — text ถูก prepend เข้าทุก chat prompt เท่านั้น ไม่มีผลต่อ code completion

Q: Update coding standards แล้วต้องทำอะไรเพิ่ม?

Code Customization: push ไป repository แล้ว Google Cloud re-index ภายใน 24 ชม.
Rules/Commands: push .vscode/settings.json แล้วทีม pull ไปใช้

Q: Code Customization indexing รองรับภาษาอะไร?

C, C++, C#, Go, Java, JavaScript, Kotlin, PHP, Python, Rust, TypeScript, Verilog, SystemVerilog และ Markdown
