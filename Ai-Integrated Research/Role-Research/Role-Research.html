3 Revised Flow (ฉบับสมบูรณ์)

🔑
Design Principles:
  • Architect ออกก่อนเสมอ — output (Tags + Scope + DDL Draft + Sec Design) กำหนดว่า DBA/Sec จะได้รับ task ไหน ตรวจอะไร
  • QA ตรวจเอกสาร ก่อน Pre-check ทำงาน — ป้องกัน DBA/Sec ทำเสียเปล่าถ้า Task โดน Reject
  • DBA/Sec สมมาตรกัน — ทั้งคู่มี Pre-check (design, ก่อน Dev) + Code Review (implementation, หลัง Dev)

Architect ใช้ decision tree นี้ตั้งแต่ Step 1 เพื่อกำหนด Tags + เขียน Review Scope ที่ระบุว่า DBA (Step 3/5) และ Sec (Step 3/5) ต้องตรวจอะไร → ส่งต่อให้ QA (Step 2) ตรวจความครบถ้วน

📌
R1 — Scope ที่ Architect ต้องระบุ:
  • 🗄️ DBA Pre  [dba-schema] → ระบุ Table ที่เพิ่ม/แก้, Column list, ความสัมพันธ์กับ Table เดิม
  • 🗄️ DBA Query  [dba-query] → ระบุ Service/Repo method ที่แตะ DB, ประเภท operation
  • 🔐 Sec Pre  [sec-design] → ระบุ Auth scheme, PII fields + masking strategy, input validation approach, endpoint authorization policy
  • 🔐 Sec Code  [sec] → ระบุ endpoint ที่แตะ Auth, PII fields ที่เกี่ยวข้อง (code-level review)
  • 📋 Std  [std-*] → ระบุขอบเขต: ไฟล์ใหม่ / แก้ไฟล์ไหน / layer ไหน
⚡
sec-design vs sec (เมื่อไหร่ใช้อะไร):
  • [sec-design] = Structural — New endpoint + Auth, New PII handling, New external integration → ต้อง Pre-check design ก่อน Dev (auto-add [sec] ด้วย)
  • [sec] only = Code-level — แก้ code ที่แตะ Auth/PII เดิม (design มีอยู่แล้ว) → ตรวจแค่ code หลัง Dev

R1

Architect ระบุ Review Scope

ทุก tag ต้องระบุ §§ + คำอธิบาย — Reviewer รู้ว่าต้องตรวจอะไร (ปิด G1)

R2

Dev QC เพิ่ม Tag ย้อนหลัง

Dev QC มีสิทธิ์เพิ่ม tag + สร้าง Follow-up Task ได้ (ปิด G2, G16)

R3

DBA Reject ให้ Corrected DDL

DBA ไม่ส่งกลับเปล่า — ให้ DDL ที่ถูกต้อง, Architect แค่ยืนยัน (ปิด G4)

R4

DBA Pre-check → Migration Script

Output = Script พร้อม run + Backward Compat + Data Impact Assessment (ปิด G12-G14)

R5

DBA-Query เทียบ Entity ↔ DDL

ตรวจทุก column ว่า Entity mapping ตรงกับ DDL ที่ Pre-check approve (ปิด G3)

R6

Sec สแกน Related Pattern

พบ vulnerability → สแกน codebase ทั้งหมด → Related Findings (ไม่ block) (ปิด G10)

R7

Std ข้ามส่วนที่ตรวจแล้ว

รับ Sec/DBA Report เป็น input → ข้าม §§ ที่ตรวจไปแล้ว ลด duplication (ปิด G6)

R8

Developer เพิ่ม Observations

Report บังคับมี Section: Security/DB/Quality Observations (กรอก "None" ถ้าไม่พบ) (ปิด G15)

R9

Observation Escalation Path

Dev QC อ่าน Observations → เพิ่ม tag / สร้าง Follow-up โดยไม่ block task ปัจจุบัน (ปิด G15-G16)

R10

Fast Track สำหรับ Hotfix

Arch → Dev → Sec (ข้าม QA+QC) เฉพาะ Production incident + Human approve (ปิด G11)

R11

DBA รองรับ Multi-tech

ตรวจได้ทั้ง Dapper (SQL), EF Core (LINQ), External (calling pattern) (ปิด G5, G9)

R12

Architect ระบุ Perf Baseline

ระบุ ก่อน/เป้าหมาย/วิธีวัด ถ้า task เป็น Performance optimization (ปิด G8)
