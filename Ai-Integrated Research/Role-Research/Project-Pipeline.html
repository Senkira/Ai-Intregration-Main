⚡ Develop — Ai-Role Pipeline (per Task)
Step 1 — 01 Solution Architect
1วิเคราะห์ Task + กำหนด Tags & Review Scope
ทำทุก task — วิเคราะห์ requirement → กำหนด tags:
[sec-design] [sec] [dba-schema] [dba-query] [std-full] [std-partial]
ทุก tag ต้องมี Review Scope (R1): ระบุ reviewer ตรวจอะไร, §§ ไหน
แนบ: DDL Draft (ถ้า schema) / Sec Design (ถ้า sec-design) / Perf Baseline (ถ้า perf, R12)
← Requirement + SAD + DB/Sec Impact Report (จาก Assessment)
→ Development_Task_{ID}.md
🏗️ 01 Architect ทุก task
Step 2 — 02 Task QA
2ตรวจเอกสาร + Tags + Scope
ทำทุก task — ตรวจ: เอกสารครบ? Tags ถูก? Review Scope ชัด?
DDL/Sec Design แนบแล้ว (ถ้ามี tag)?
REJECT → กลับ Step 1 (ป้องกัน Pre-check ทำเสียเปล่า)
APPROVE → มี [dba-schema] / [sec-design]? → Step 3 : ไม่มี → Step 4
← Development_Task_{ID}.md
→ Task_QA_Report_{ID}.md (APPROVE / REJECT)
🔬 NotebookLM — Chat + Citation
อัปโหลด task doc + SAD + Impact Reports + Task Template → ถาม "เอกสารครบไหม?"
citation ชี้ว่าข้อมูลไหนยัง ขาด/ไม่ตรง กับ reference — ไม่ต้อง cross-check manual
📋 02 Task QA ทุก task 🚩 Gate 🔬 NotebookLM
Step 3 — Pre-check Gate (ก่อน Dev · conditional · parallel)
3🗄️ DBA Pre-check
เมื่อมี [dba-schema] — ตรวจ DDL Draft:
Schema naming · Index design · Backward compat · Data impact
APPROVE → output Migration Script (R4)
REJECT → ส่ง Corrected DDL กลับ Step 1 (R3)
← DDL Draft + Review Scope จาก Step 1
→ DBA_Precheck_{ID}.md + Migration_Script_{ID}.sql
🗄️ DBA Pre-check if [dba-schema]
3🔐 Sec Pre-check
เมื่อมี [sec-design] — ตรวจ Sec Design:
Auth scheme · PII masking strategy · Input validation approach · Authz policy
APPROVE → Approved Design เป็น spec ให้ Dev
REJECT → ส่ง Corrected Design กลับ Step 1
← Sec Design + Review Scope จาก Step 1
→ Sec_Precheck_{ID}.md + Approved Auth/PII Design
🔐 Sec Pre-check if [sec-design]
Step 4 — 03 Developer
4Implement + Observations
ทำทุก task — Code ตาม Task + Approved Design จาก Pre-check (ถ้ามี)
บังคับมี Observations section (R8):
Security observations / DB observations / Quality observations
กรอก "None" ถ้าไม่พบ — ห้ามละเว้น
← Development_Task_{ID}.md + Approved DDL/Sec Design (ถ้ามี)
→ Code Changes + Development_Report_{ID}.md
💻 03 Developer ทุก task
Step 5 — Parallel Review Gate (หลัง Dev)
504 Dev QC
ทำทุก task — ตรวจ: Scope/Logic ถูกตาม Task?
อ่าน Observations → Triage (R9):
พบ issue → เพิ่ม tag ย้อนหลัง (R2) / สร้าง Follow-up Task
REJECT → กลับ Step 4
← Code + Development_Report_{ID}.md
→ Dev_QC_Report_{ID}.md
✅ 04 Dev QC ทุก task
5🔐 Sec Code Review
เมื่อมี [sec] หรือ [sec-design]
ตรวจ code จริง: Auth/Validation impl ตรง approved design?
Error leak? PII in code? Secret? Test data bypass?
+ Related Pattern Scan ทั้ง codebase (R6)
Primary §§: §3.6, §3.21-3.23, §7, §8.6, §9, §15, §16.4, §18-§19
← Code + Review Scope + Approved Sec Design (ถ้ามี)
→ Security_Review_{ID}.md
🔐 Sec Code Review if [sec] / [sec-design]
5🗄️ DBA Query Review
เมื่อมี [dba-query]
ตรวจ code จริง: Query perf? N+1? Transaction? Audit log?
Entity↔DDL match (R5)?
รองรับ EF Core + Dapper + External System (R11)
Primary §§: §3.16-3.20, §3.24-3.27, §17
← Code + Review Scope + Approved DDL (ถ้ามี)
→ DBA_Review_{ID}.md
🗄️ DBA Query Review if [dba-query]
Step 6 — Coding Standard Reviewer (conditional)
6📋 Std Reviewer — 295 เกณฑ์
เมื่อมี [std-full] หรือ [std-partial]
ตรวจ: Naming §2 · Structure §1 · DI §11 · Testing §12
API Design §10,§13 · YAGNI §20 · Format §14
ข้ามส่วนที่ Sec/DBA ตรวจแล้ว (R7) — รับ report มาอ้างอิง
Primary §§: §1-§2, §3.1-3.15, §3.28-3.30, §4-§6, §8, §10-§14, §16, §20-§26
🔬 NotebookLM — Source-grounded Chat + Citation
อัปโหลด 295 เกณฑ์ (Coding Standard ฉบับเต็ม) + code + Sec/DBA Reports →
ถาม "code นี้ขัดเกณฑ์ข้อไหน?" → citation ชี้กลับ § จริง ไม่ hallucinate เลข §
⚡ ทำไมต้อง NLM: 295 เกณฑ์ = ~192KB · Copilot อาจจำ § ผิด · NLM cite exact quote จาก §
← Code + Sec/DBA Review Reports (ถ้ามี)
→ Std_Review_{ID}.md
📋 Std Reviewer if [std-full] / [std-partial] 🔬 NotebookLM
✅ MERGE READY — ทุก Review ผ่าน → merge to main
